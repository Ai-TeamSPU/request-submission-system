# แผนการดำเนินงาน: ระบบกำหนดสิทธิ์เริ่มต้นและการจัดการบทบาทโดยผู้ดูแลระบบ (Admin Role Management)

ขยายความสามารถระบบจัดการสิทธิ์การเข้าถึงให้ควบคุมผ่านฐานข้อมูลจริง (Database-driven Permission) แทนการตรวจสอบด้วยโดเมนอีเมลเบื้องต้น โดยจะประกอบด้วย:
1. การบันทึกสิทธิ์ผู้ใช้งานใหม่ที่ล็อกอินครั้งแรกให้เป็น **อาจารย์ (`teacher`)** เป็นค่าเริ่มต้นโดยอัตโนมัติ
2. การสร้างหน้าจอผู้ดูแลระบบ (Admin Panel) ในเมนู **ข้อมูลหลัก** เพื่อตรวจสอบรายชื่อผู้ใช้งานทั้งหมด และสามารถปรับเปลี่ยนบทบาทผู้ใช้งาน (อาจารย์, คณบดี, ผอ., Admin) ได้โดยตรงผ่านระบบหน้าบ้าน

---

## 🎨 การจัดวางและการแสดงผลในหน้าข้อมูลหลัก (Master Data UI Changes)

ในหน้าจอเมนู **ข้อมูลหลัก (Master Data)** ของฝั่งผู้ดูแลระบบ (Admin) จะเพิ่มแท็บหรือตาราง **"การจัดการบทบาทและสิทธิ์ผู้ใช้งาน"** ซึ่งมีรายละเอียดดังนี้:

```
+------------------------------------------------------------------------------------------+
| 🗃️ การจัดการข้อมูลหลักกลางของระบบ (Master Data Management)                                         |
| +--------------------------------------------------------------------------------------+ |
| | [แท็บ: ข้อมูลผู้ใช้งานในระบบ 🌟]   [แท็บ: รายชื่อคณะ]   [แท็บ: ประเภทปัญหา]                     | |
| +--------------------------------------------------------------------------------------+ |
| | อีเมลผู้ใช้                        | บทบาทปัจจุบัน              | เปลี่ยนสิทธิ์การเข้าถึง            | |
| |----------------------------------+-------------------------+-------------------------| |
| | teacher@university.ac.th         | 🧑‍🏫 อาจารย์              | [ เลือกบทบาท (Dropdown) v] | |
| | dean@university.ac.th            | 🏛️ คณบดี                 | [ เลือกบทบาท (Dropdown) v] | |
| | admin@university.ac.th           | 🔑 ผู้ดูแลระบบ (Admin)     | [ เลือกบทบาท (Dropdown) v] | |
| +--------------------------------------------------------------------------------------+ |
+------------------------------------------------------------------------------------------+
```

## 🛠️ แผนการปรับปรุงไฟล์โค้ด (Refactoring Plan)

### 1. ปรับปรุงโมดูลจัดการฐานข้อมูล ([src/utils/database.js](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/request%20submission%20system/src/utils/database.js))
- ปรับโครงสร้างข้อมูลสมาชิกระบบใน `localStorage` ให้เก็บแบบรายวัตถุ `{ email, role }`
- ปรับฟังก์ชัน `saveEmail(email)`:
  - หากอีเมลมีอยู่ในระบบแล้ว ให้คืนค่าข้อมูลเดิม
  - หากยังไม่มีในระบบ ให้บันทึกใหม่และกำหนดค่าเริ่มต้นให้เป็นสิทธิ์อาจารย์ (`teacher`) ทันที
  - (รองรับการดึงข้อมูล `users_list` บน Supabase จริงด้วย)
- เพิ่มฟังก์ชัน `getUsersList()`: ดึงรายชื่อผู้ใช้ทั้งหมดในระบบมาแสดงในหน้าตารางข้อมูลหลัก
- เพิ่มฟังก์ชัน `updateUserRole(email, role)`: อัปเดตสิทธิ์บทบาทของผู้ใช้งานคนนั้น ๆ ลงฐานข้อมูล Supabase (`users_list`) หรือ localStorage

### 2. ปรับปรุงหน้าจอควบคุมหลัก ([src/App.jsx](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/request%20submission%20system/src/App.jsx))
- อัปเดตการตรวจสอบสิทธิ์ล็อกอิน: เมื่อผู้ใช้กรอกอีเมลเข้ามา ระบบจะไปดึงสิทธิ์จริง (`role`) จากฐานข้อมูลแทนการใช้ Regex เช็คโดเนมอีเมลเดิม
- ปรับปรุงการเรนเดอร์หน้าจอ **ข้อมูลหลัก (renderMasterData)** ของ Admin:
  - เพิ่มการดึงข้อมูล `getUsersList()` บน React State
  - แสดงผลตารางผู้ใช้งานทั้งหมด
  - เพิ่ม Dropdown เลือกบทบาทข้างท้ายแต่ละแถวข้อมูล เมื่อแอดมินเปลี่ยนค่า จะเรียกใช้ฟังก์ชัน `db.updateUserRole` ทันทีและรีเฟรชตารางแบบเรียลไทม์

### 3. อัปเดตเอกสารคู่มือเชื่อมต่อ ([คำอธิบาย/supabase_integration.md](file:///c:/Users/mmath/.gemini/antigravity-ide/scratch/request%20submission%20system/คำอธิบาย/supabase_integration.md))
- เพิ่มรายละเอียดฟังก์ชันและวิธีการเขียนสคริปต์สำหรับการจัดการข้อมูลผู้ใช้งานและการอัปเดตสิทธิ์บน Supabase (`updateUserRole`)

## 📋 แผนการตรวจสอบและทดสอบ (Verification Plan)
- **การทดสอบสิทธิ์เริ่มต้น:** ล็อกอินด้วยอีเมลใหม่ที่ยังไม่เคยอยู่ในระบบ (เช่น `new.teacher@university.ac.th`) และตรวจสอบว่าระบบกำหนดสิทธิ์เริ่มต้นเป็น **อาจารย์ (teacher)** พร้อมเปิดเฉพาะเมนูของอาจารย์เท่านั้น
- **การทดสอบเปลี่ยนสิทธิ์โดย Admin:**
  1. ล็อกอินเข้าใช้งานด้วยบัญชีผู้ดูแลระบบ (`admin@university.ac.th`)
  2. ไปที่เมนู **ข้อมูลหลัก**
  3. ปรับสิทธิ์ของบัญชีทดสอบอื่น (เช่น ปรับสิทธิ์ของ `teacher@university.ac.th` ให้เป็น `dean`)
  4. ทำการออกจากระบบของ Admin แล้วนำบัญชีทดสอบที่ถูกเปลี่ยนสิทธิ์นั้นมาล็อกอินใหม่
  5. ตรวจสอบว่าเมนูแถบข้างได้สลับเป็นหน้าจอคิวพิจารณาของคณบดีตามที่ได้รับสิทธิ์ใหม่เรียบร้อยแล้วหรือไม่
